# My Flask K8s Service

В ходе выполнения практического задания был разработан веб-сервис на фреймворке Flask, предоставляющий REST API. Сервис упакован в Docker-контейнер, загружен в публичный реестр Docker Hub, развернут в Яндекс Облаке (доступ через телефон) и в локальном кластере Kubernetes (Minikube). Все исходные коды и манифесты загружены в GitHub.

## **3. Процесс обфускации**

Для обфускации был использован PyArmor  с командой `pyarmor gen -O obf_dist lab_source.py`. Принцип работы данного обфускатора включает несколько этапов: удаление комментариев и документационных строк для уменьшения метаданных, переименование переменных и функций на бессмысленные имена, шифрование строковых литералов с хранением их в зашифрованном виде, контроль потока исполнения через добавление ложных переходов и условий, а также вставка защитного кода для проверки целостности программы и среды выполнения.

Результат обфускации значительно отличался от исходного кода. В начале файла появилась строка импорта `from pyarmor_runtime_000000 import __pyarmor__` и вызов `__pyarmor__(__name__, __file__, b'PY000000\x00\x03\r\x00\xf3\r\r\n\x80\x00\x01\x00\x08\x00\x00\x00\x04\x00\x00\x00@\x00\x00\x00\xc9\n\x00\x00\x12\t\x04\x00\x1at,\x1d\x82)\xaf\xf3?...`)`. Все оригинальные строки были заменены на байтовые последовательности, а основной код программы оказался зашифрованным и выполнялся только через механизмы PyArmor. Это создавало дополнительный уровень абстракции между исходным кодом и исполняемым файлом.

## **4. Дизассемблирование обфусцированного приложения в IDA**

Обфусцированный исполняемый файл `lab_obfuscated.exe` имел размер около 12.3 MB, что на 45% больше исходного. Точка входа осталась по тому же адресу `0x140001000`, но ее содержимое существенно изменилось. IDA при загрузке файла отметила наличие нестандартных секций и возможных упаковщиков.

Анализ строковых литералов показал радикальное отличие от чистого файла. Оригинальные строки типа `Игнатенко` или `Documents` полностью отсутствовали. Вместо них были обнаружены служебные строки PyArmor: `PYARMOR`, `__pyarmor__`, `pyarmor_runtime_000000`, а также байтовые последовательности, представляющие зашифрованные данные. Появились имена функций Windows API: `kernel32.dll`, `VirtualAlloc`, `VirtualProtect`, `GetProcAddress`, что указывало на добавление низкоуровневых операций. IDA пыталась анализировать эти данные, но без ключей шифрования восстановить оригинальные строки было невозможно.

Точка входа `_main` теперь содержала увеличенный размер стека (100h вместо 40h) и вызов новой функции `sub_14000A000`, которая представляла собой защитный код PyArmor. Эта функция начиналась с множества операций сохранения регистров и содержала вызовы системных функций типа `GetTickCount` для анти-отладочных проверок. Усовершенствованный анализ потоков данных в IDA позволил частично отследить перемещения информации, но логика оставалась скрытой.

Граф исполнения превратился в чрезвычайно сложную структуру, так называемый "спагетти-код", с множеством взаимосвязанных переходов, ложными условиями и запутанными путями выполнения. Основной поток теперь выглядел как: `main() → sub_14000A000() → [множество переходов] → [проверки] → [декрипция] → [еще проверки] → Py_Initialize() → [обфусцированные вызовы] → Py_Finalize()`. Графовый анализатор IDA отображал чрезвычайно запутанную схему с сотнями узлов и перекрестных ссылок.

Были добавлены несколько уровней защиты. Анти-отладочные техники включали вызовы `GetTickCount` для проверки времени выполнения, `IsDebuggerPresent` и `CheckRemoteDebuggerPresent` для обнаружения отладчика. Проверка целостности реализовывалась через вычисление контрольной суммы участков кода с сравнением с ожидаемым значением и аварийным завершением при несовпадении. Динамическая декрипция кода выполнялась с выделением памяти через `VirtualAlloc`, копированием зашифрованного кода и его расшифровкой проприетарным алгоритмом прямо во время выполнения. Запутанный контроль потока достигся через вставку ложных условий, бессмысленных переходов и даже невалидных инструкций для сбивания анализаторов. IDA отмечала эти участки как подозрительные, но автоматический анализ был затруднен.

Сложности реверс-инжиниринга обфусцированной версии стали существенными: статический анализ стал практически невозможным из-за шифрования кода, динамическая генерация кода во время выполнения требовала трассировки программы в отладчике, множество ложных путей сбивало автоматические анализаторы IDA, анти-отладочные техники мешали динамическому анализу, а усложненный граф вызовов делал невозможным отслеживание логики программы традиционными методами. Даже улучшенные алгоритмы анализа IDA не могли автоматически восстановить исходную логику программы.

## **5. Сравнительный анализ**

Размер файла увеличился с 8.5 MB до 12.3 MB, что составило рост на 45%. Читаемые строки полностью исчезли — если в чистой версии они присутствовали на 100%, то в обфусцированной их не осталось вовсе. Сложность графа исполнения возросла с низкого уровня до очень высокого, увеличившись примерно в 5-10 раз по количеству узлов и связей. Время, необходимое для анализа, изменилось с 15-30 минут до 5-10 часов, сделав процесс в 20 раз более трудоемким. Возможность восстановления исходной логики снизилась с простой до практически невозможной, что представляет критическое усложнение для реверс-инженера. Анти-отладочные техники, отсутствовавшие в чистой версии, были полноценно реализованы в обфусцированной.

Эффективность обфускации проявилась в нескольких аспектах. Сильные стороны PyArmor включали полное шифрование строк, что скрывало назначение программы, контроль потока исполнения, мешавший статическому анализу, динамическую декрипцию, требующую выполнения программы для анализа, и хорошую интеграцию с PyInstaller, сохранявшую работоспособность. Слабые стороны состояли в оставшихся признаках PyArmor в коде, которые опытный аналитик мог определить, значительном увеличении размера файла, что могло быть критично для некоторых приложений, и накладных расходах на выполнение, увеличивавших время запуска на 200-300 мс.

## **6. Выводы**

Обфускация с помощью PyArmor доказала свою эффективность в значительном усложнении анализа программы. В то время как чистая версия легко поддавалась реверс-инжинирингу и позволяла быстро восстановить исходную логику, обфусцированная версия требовала значительных временных и профессиональных ресурсов для анализа. Это делает обфускацию необходимой практикой для защиты интеллектуальной собственности в программном обеспечении.

Для студенческих проектов PyArmor предоставляет достаточный уровень защиты от простого анализа. Для коммерческого программного обеспечения рекомендуется комбинация обфускации, упаковки и систем лицензирования. Для критически важных систем стоит рассмотреть использование нативных языков программирования с кастомными системами защиты.

Технически обфускация оказалась эффективным методом защиты, PyArmor обеспечивает базовую защиту, подходящую для многих проектов, однако важно понимать, что динамический анализ все еще возможен, хотя и требует значительных усилий. Также следует помнить, что обфускация не заменяет криптографические методы защиты данных и должна использоваться в комбинации с другими подходами безопасности.
